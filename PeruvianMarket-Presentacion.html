Motivación

Apostar entre amigos exige confianza. La criptografía la reemplaza.

🔑 El problema de custodia

¿Quién guarda el dinero? Nadie debería poder mover tus fondos sin tu autorización. Solución: cada usuario genera su propio par de llaves; toda transferencia exige su firma ECDSA.

⚖️ El problema del árbitro

¿Quién decide si "Perú clasifica al Mundial" ocurrió? Solución: un oráculo con llave Ed25519 firma cada resolución — cualquiera puede verificarla.

📜 El problema del historial

¿Cómo pruebo que una apuesta existió? Solución: ledger de transacciones auditable con firmas persistidas, consultable por todos.

Producto: mercados Sí/No y multi-opción con AMM (x·y=k), casino con 7 juegos, wallet CHC, minería, tareas — todo operando sobre las mismas primitivas criptográficas.

lib/crypto.ts — derivación real

function deriveAddress(pubKeyBytes) { const hash160 = ripemd160(sha256(pubKeyBytes)); const versioned = [0x35, ...hash160]; // "P" const checksum = sha256(sha256(versioned)) .slice(0, 4); return base58Encode([...versioned, ...checksum]); }

Ejemplo real de la plataforma

addr NLYTpsuc7xPNerk3ETAwJNAaqLQ95wHx94
pub  031b6d7b79dc52c1…873cc32a5f (33 B)

El checksum de 4 bytes detecta errores de tipeo: una dirección alterada falla la verificación antes de enviar fondos. Mismo esquema que Bitcoin (P2PKH), con byte de versión propio.

Asimétrica I · Firma de transacciones

Cada transferencia lleva una firma ECDSA verificada en el servidor

Cliente — firma (lib/crypto.ts)

// JSON canónico: claves ordenadas → // mismo hash en cliente y servidor const msg = canonicalJSON(txData); const hash = sha256(encode(msg)); const sig = secp.sign(hash, privKey); return sig.toCompactRawBytes(); // 64 B

Servidor — verificación (api/wallet/transfer)

if (!verifySignature(tx, sig, pubKey)) return 401 "Firma inválida"; // pubKey debe pertenecer a la wallet if (!walletOwns(from, pubKey)) return 401 "Wallet no autorizada"; // anti-replay: nonce monotónico if (nonce !== currentNonce + 1) return 400 "Nonce inválido";

Mensaje firmado: {type, from, to, amount, nonce, timestamp} — cualquier alteración de un solo campo invalida la firma.

Parámetros (lib/crypto.ts)

cifrado
AES-GCM 256 bits — autenticado: cifra y sella integridad en una operación
KDF
PBKDF2-SHA256, 600 000 iteraciones (OWASP 2023) · formato versionado v2: con migración transparente desde blobs legacy
salt
16 bytes CSPRNG, único por cifrado — evita rainbow tables
IV/nonce
12 bytes CSPRNG, único por cifrado — nunca se reutiliza con la misma llave
motor
WebCrypto (crypto.subtle) — nativo del navegador

Propiedad clave del diseño

El servidor nunca ve la llave privada: ni en claro ni cifrada. El blob vive solo en el navegador del usuario. Sin la contraseña maestra, el blob es inútil — GCM además detecta cualquier manipulación del ciphertext (tag de autenticación de 128 bits).

Asimétrica II · Ed25519

El oráculo firma cada resolución de mercado

lib/oracle.ts — server-only

async function signResolution(marketId, outcome, ts) { const msg = `${marketId}:${outcome}:${ts}`; const sig = await ed.signAsync( encode(msg), ORACLE_PRIVATE_KEY // solo en .env del server ); return hex(sig); // persistida junto al mercado }

¿Por qué Ed25519 y no ECDSA otra vez?

  • Firmas deterministas — sin nonce aleatorio por firma; elimina la clase de fallas que rompió consolas y wallets con ECDSA mal implementado
  • Verificación pública — la llave pública del oráculo se expone; cualquier usuario puede validar que una resolución es auténtica
  • Separación de dominios — usuarios firman con secp256k1, el árbitro con Ed25519: una llave comprometida no cruza roles

Flujo: mercado cierra → oráculo firma marketId:YES:1720000000 → firma + timestamp persistidos → pagos distribuidos → cualquiera audita la firma contra la llave pública.

🔒 Privacidad real (simétrica + acuerdo de claves)

Ambas partes derivan la misma clave AES-256 sin intercambiarla jamás: ECDH sobre las llaves secp256k1 de sus wallets + SHA-256. El servidor almacena solo ciphertext — los términos de la apuesta son ilegibles para todos excepto los dos apostadores.

✍️ No repudio de punta a punta

  • Las firmas de creación y aceptación se anclan a SHA-256(ciphertext): ambos firmaron exactamente los mismos términos
  • Cada veredicto lleva firma ECDSA propia — nadie puede negar su voto
  • Si el oráculo interviene, su resolución queda firmada con Ed25519 y es auditable

Escrow: ambos depósitos quedan bloqueados hasta resolución — pago con claim condicional (imposible cobrar dos veces).

Auditoría · Blockchain

Bloques encadenados con SHA-256 — integridad verificable en un clic

lib/blockchain.ts — hash de bloque

block_hash = SHA-256( número | prev_hash | tx₁ id:tipo:from:to:monto ; tx₂ … tx₅ ) // alterar cualquier tx sellada o bloque // rompe la cadena desde ese punto

Cada 5 transacciones se sellan en un bloque cuyo hash cubre el contenido de las transacciones y el hash del bloque anterior — desde el bloque génesis (0×64).

🛡️ Verificación pública de integridad

El explorador incluye "Verificar cadena": el endpoint /api/blockchain/verify recomputa el hash de todos los bloques y valida el encadenamiento.

  • Transacción alterada en la BD → hash no coincide → reporta el bloque exacto
  • Transacción eliminada → detectada como faltante
  • Bloque reordenado → prev_hash rompe la cadena

demo en vivo: editar un monto en Supabase → verificar → cadena rota en bloque #N

Doble garantía: firmas ECDSA por transacción (autenticidad del emisor) + encadenamiento por bloque (inmutabilidad del historial).

Nonce monotónico

Cada cuenta lleva un contador. El nonce va dentro del mensaje firmado: no se puede editar sin invalidar la firma.

Doble verificación de identidad

La firma prueba posesión de la llave; la BD confirma que esa llave pública corresponde a la dirección emisora.

Validación de fondos

Saldo verificado server-side con cliente admin (bypass de RLS solo en backend) — el cliente jamás escribe balances.

Limitaciones y trabajo futuro

Lo que mejoraríamos con más tiempo

Endurecimiento criptográfico

  • PBKDF2 elevado a 600k iteraciones con formato versionado — hecho ✓
  • Hash chaining real de bloques + verificación pública — hecho ✓
  • Migrar KDF a Argon2id (resistente a GPU/ASIC)
  • Rotación de la llave del oráculo con publicación de llaves históricas

Superficie de aplicación

  • Firmar también las apuestas de casino (hoy: custodial, autorizado solo por sesión JWT)
  • Rate limiting en endpoints de firma para frenar fuerza bruta online
  • Análisis estático de seguridad integrado a CI (Semgrep / CodeQL) — requisito de la entrega 2

Criterio de priorización: primero lo que amplía garantías criptográficas (encadenamiento, Argon2id), después lo operacional.
